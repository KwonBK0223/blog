> **Paper**: BHFVAL: Blockchain-Enabled Hierarchical Federated Variational Autoencoder Framework for Secure Intrusion Detection in Vehicular Networks — G. E. Visuvanathan, Md S. Sayeed, S. Yogarayan, *Scientific Reports*, vol. 15, art. 45742, Dec. 2025. [doi:10.1038/s41598-025-28560-y](https://doi.org/10.1038/s41598-025-28560-y)

## TL;DR

A comprehensive framework for intrusion detection in vehicular networks (IoV) that centers on a Hierarchical Federated Variational Autoencoder (HFVAL) and layers on blockchain-based validation (RBFT), lightweight edge computing (LEC), metaheuristic hyperparameter optimization (OOA), and a lightweight secure communication protocol (LSCP). It aims for scalable, privacy-preserving, resource-efficient FL, strengthening security, efficiency, and flexibility all at once.

## Background

As clients multiply, the cloud server becomes a bottleneck. HFL alleviates this with a cloud–edge–client structure: multiple edges receive local updates first, and the edges forward parameters to the cloud server. This lets more clients participate in the framework and reduces computational load.

The system in this paper has three tiers: **edge nodes, fog-level RSUs, and a cloud center**. The edge is the compute resource closest to the vehicles, performing computation, offloading, and partial aggregation nearby; the fog is an intermediate tier above the edge that gathers updates from multiple vehicles or edges and performs intermediate aggregation.

The paper also emphasizes handling **zero-day attacks** — attacks that, from the defender's perspective, are as yet unknown, insufficiently represented in the training data, or not covered by any detection rules.

## Method

The framework's main components are as follows.

- **RBFT (Reputation-based Byzantine Fault Tolerance) blockchain**: protects model aggregation and guarantees tamper-proof parameter storage. The blockchain is not for storing raw data — it serves as a tamper-resistant shared ledger that guarantees the integrity of model updates.
- **Lightweight Edge Computing (LEC)**: dynamically offloads computation from vehicle nodes to nearby edge servers to minimize latency.
- **Osprey Optimization Algorithm (OOA)**: used for hyperparameter optimization to improve convergence under real-time constraints.
- **LSCP-based ECHE**: a lightweight secure communication protocol enabling encrypted V2X communication with minimal computational overhead.
- **Class-imbalance handling**: SMOTE-based, combining majority-class undersampling with minority-class oversampling. SMOTE does not fabricate "fake attack data" at random — it synthesizes new minority-like samples within the feature space of existing minority-class samples.

### Overall training flow

One communication round proceeds as follows.

1. **Data preprocessing**: each client removes unnecessary features and performs one-hot encoding, min-max scaling, and missing-value handling on its local data.
2. **Initial global model preparation**: the top-level cloud server prepares the initial model or the previous round's global model.
3. **Cloud → fog/RSU/edge transfer**: rather than talking to every vehicle directly, the cloud pushes the model down to the intermediate aggregator tier.
4. **Edge → vehicle distribution**: each edge/RSU distributes the global model to its attached vehicle nodes.
5. **VAE encoding**: vehicles never share raw data directly; the VAE encoder transforms it into latent feature representations.
6. **Local IDS model update**: each vehicle performs local gradient descent on its local data or latent representations.
7. **OOA hyperparameter tuning**: learning rate, batch size, momentum, etc. are adjusted dynamically.
8. **LEC placement**: training and aggregation workloads are assigned to the appropriate tier — vehicle, edge, or fog — based on each vehicle's compute resources and network conditions.
9. **Encrypted transmission**: local updates are sent to the RSU/fog/edge over secure V2X communication using LSCP and ECHE.
10. **Update collection at intermediate aggregators**: local updates are gathered from multiple vehicle clients.
11. **RBFT validation**: malicious or low-quality updates are filtered based on reputation scores, signatures, and update consistency.
12. **Edge/fog-level aggregation**: reputation-weighted aggregation is performed over validated updates only. Mathematically, it is close to FedAvg-style weighted averaging.
13. **Blockchain recording**: model update hashes, reputation scores, and signatures are stored on the blockchain ledger to prevent tampering.
14. **Fog/edge → cloud transfer**: each intermediate aggregator uploads its aggregated model update to the cloud.
15. **Cloud global aggregation**: the edge/fog aggregated models are merged into a new global model.
16. **Redistribution**: the new global model flows cloud → fog/RSU/edge → vehicle.
17. The next communication round repeats.

In summary, the pipeline is: **vehicle-local VAE-based representations/updates → RSU/fog reputation-weighted aggregation → blockchain/RBFT validation → cloud global aggregation → redistribution**.

## Experiments

- **Datasets**: UNSW-NB15, CIC-IDS-2017
- Latency and resource-utilization analyses verify the low communication overhead of LSCP/ECHE.

## Takeaways

- A systems paper with many moving parts, integrating HFL + VAE + blockchain + an optimization algorithm + secure communication into one framework.
- The aggregation itself is reputation-weighted FedAvg, and the blockchain is used narrowly for update-integrity validation.
- Several points remain unclear, however: whether the local training loss is the VAE loss or a classification loss, whether the task is classification or anomaly detection, and how substantially the proposed aggregation formula actually differs from FedAvg — none of these are clearly stated in the paper.
