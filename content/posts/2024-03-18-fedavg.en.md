> **Paper**: Communication-Efficient Learning of Deep Networks from Decentralized Data — H. Brendan McMahan, Eider Moore, Daniel Ramage, Seth Hampson, Blaise Agüera y Arcas, *AISTATS 2017*. [arXiv:1602.05629](https://arxiv.org/abs/1602.05629)

## TL;DR

- Mobile devices hold a wealth of data useful for training models, but it is privacy-sensitive and large in volume, making it impractical to collect in a data center.
- The paper proposes leaving data on each device and **training a shared model by aggregating only locally-computed updates**, an approach it names **Federated Learning**.
- The core algorithm, **FedAvg**, combines local SGD on each client with model averaging on the server, reducing the required communication rounds by **10–100x** compared to synchronous SGD (FedSGD).
- Its robustness to unbalanced and non-IID data distributions is demonstrated empirically across five model architectures and four datasets.

## Background

### Why Federated Learning

Federated learning is a setting in which a loose federation of participating devices (clients) solves a learning problem under the coordination of a central server. Each client computes an update using only its local dataset, and the server updates the global model by aggregating only the received updates.

The ideal problems for federated learning have the following properties.

1. Training on real data from mobile devices offers a clear advantage over training on proxy data available in a data center.
2. The data is privacy-sensitive or large enough that transmitting it to a data center is undesirable.
3. For supervised tasks, labels can be inferred naturally from user interaction.

From a privacy standpoint, the transmitted information is the **minimal update** necessary to improve a particular model; the updates themselves are ephemeral, contain less information than the raw training data, and require no provenance tracking (this is not a complete guarantee, as discussed later).

### Properties of Federated Optimization

The authors call the optimization problem implicit in federated learning **federated optimization**, distinguishing it from standard distributed optimization along four axes.

- **Non-IID**: each client's data is based on a particular user's device usage and is not representative of the population distribution.
- **Unbalanced**: usage varies across users, so the amount of local data differs widely.
- **Massively distributed**: the number of clients far exceeds the average number of examples per client.
- **Limited communication**: mobile devices are frequently offline, and connections are slow or expensive.

The objective is the standard ERM objective restructured at the client level.

$$f(\omega) = \sum_{k=1}^{K} \frac{n_k}{n} F_k(\omega), \qquad F_k(\omega) = \frac{1}{n_k} \sum_{i \in P_k} f_i(\omega)$$

Here $K$ is the number of clients, $P_k$ is the set of data indices on client $k$, and $n_k = |P_k|$.

In data-center optimization, **computation cost** dominates; in the federated setting, **communication cost** does. The goal is therefore inverted — spend extra computation to reduce communication rounds. Two levers exist: increased parallelism (more participating clients) and **increased computation per client**.

## Method

### From FedSGD to FedAvg

The baseline, **FedSGD**, has each client compute the average gradient over its entire local dataset and send it to the server, which takes a single weighted-aggregate step.

$$\omega_{t+1} \leftarrow \omega_t - \eta \sum_{k=1}^{K} \frac{n_k}{n} g_k, \qquad g_k = \nabla F_k(\omega_t)$$

Computation per round is small, but the frequent updates require a lot of communication.

**FedAvg** has each client iterate the local update multiple times before averaging. The algorithm is controlled by three parameters.

- $C$: the fraction of clients participating in each round
- $E$: the number of training epochs each client runs over its local data per round
- $B$: the local minibatch size

The server loop is as follows. In each round $t$, select $m = \max(C \cdot K, 1)$ clients at random; each client $k$ starts from the current global parameters $\omega_t$, splits its local data into batches of size $B$, runs $\omega \leftarrow \omega - \eta \nabla \ell(\omega; b)$ for $E$ epochs, and sends $\omega_{t+1}^k$ to the server. The server updates with a data-size-weighted average.

$$\omega_{t+1} \leftarrow \sum_{k \in S_t} \frac{n_k}{m_t}\, \omega_{t+1}^k, \qquad m_t = \sum_{k \in S_t} n_k$$

With $B = \infty$ (full batch) and $E = 1$, this reduces to FedSGD. It is not obvious why averaging model parameters should work for non-convex problems, but the authors show empirically that averaging models initialized from the same starting point works well.

## Experiments

**MNIST (2NN, CNN)** — Experiments cover both IID and non-IID (digit-class shard) partitions. Counting the communication rounds needed to reach target accuracy (97%/99%), FedAvg ($C=0.1$, $E=5$, $B=10$) on the CNN reaches the target in 18 rounds in the IID setting — about 2.8x faster than FedSGD (387 rounds) — and over 4.6x faster in the non-IID setting. Increasing the client fraction $C$ and the amount of local computation ($E$↑, $B$↓) accelerates convergence.

**Shakespeare (LSTM, character-level language model)** — In a naturally non-IID setting where data is partitioned by character role, FedAvg ($E=5$, $B=10$) reaches 54% accuracy in 13.0x fewer rounds than FedSGD in the IID case and **95.3x fewer in the non-IID case**. FedAvg's robustness is, if anything, more pronounced on non-IID data.

**CIFAR-10 (CNN)** — To reach 80% target accuracy: SGD takes 18,000 rounds, FedSGD 3,750, and FedAvg **280** (a 64.3x reduction). At the 85% target, only FedAvg reaches it, in 2,000 rounds.

In summary, four conclusions.

- **Robust performance**: strong even under the non-IID, unbalanced distributions common in real data.
- **Communication efficiency**: dramatically fewer communication rounds to convergence, addressing the key constraint of the federated setting.
- **Accuracy**: higher accuracy (better generalization) than FedSGD across multiple datasets.
- **Scalability**: scales across a range of network sizes and data distributions.

## Takeaways

With the simple idea of "compute more locally, communicate less often," FedAvg brought federated learning to a practical level, and it became the starting point and standard baseline for the many FL algorithms that followed (FedProx, SCAFFOLD, and others).

Its limitations are just as clear, however. Subsequent work — Deep Leakage from Gradients (Zhu et al., 2019) — showed that from shared gradients one can determine whether a sample with a particular property is in the batch, train a generative model to produce similar data, or even reconstruct the training data itself. In other words, **sharing only updates does not provide complete privacy protection**. Several directions are being studied to address this.

1. **Differential Privacy (DP)**: adding noise so that individual data points cannot be distinguished
2. **Secure Multi-party Computation (SMPC)**: jointly computing a function while keeping inputs private
3. **Homomorphic Encryption (HE)**: computing directly on encrypted data
4. **Model Pruning / Quantization**: reducing model size and complexity to mitigate the risk of data reconstruction from outputs

As the paper that opened up the field of federated learning itself, it is the first classic anyone studying FL should read.
