> **Paper**: Distributed cross-learning for equitable federated models — privacy-preserving prediction on data from five California hospitals — Kuo et al., *Nature Communications*, 2025. [DOI: 10.1038/s41467-025-56510-9](https://www.nature.com/articles/s41467-025-56510-9)

## TL;DR

- 의료 데이터는 여러 기관에 **수평(horizontal)** 으로도, **수직(vertical)** 으로도 흩어져 있으며, 법·윤리·정책적 제약 때문에 원본 데이터의 이동이 사실상 불가능하다.
- D-CLEF는 환자 레벨 데이터를 절대 외부로 내보내지 않으면서, **중앙 서버 없이** peer-to-peer 방식으로 수평·수직 분할 데이터를 모두 학습할 수 있는 완전 분산 협력 학습 프레임워크다.
- 학습 과정의 메타데이터는 permissioned **블록체인**에, 모델 파라미터는 **분산 파일 시스템(DFS)** 에 기록하여 감사 가능성(auditability)과 확장성을 동시에 확보한다.
- 캘리포니아 5개 병원의 COVID-19 사망 예측, 수술 후 장기 입원 예측, 심근경색 예측이라는 실제 의료 문제에서 **중앙집중 학습과 통계적으로 유의미한 차이가 없는 성능**을 달성했다.
- 새로운 VFL 알고리즘이나 암호학적 보장을 제안하는 논문이 아니라, 의료 기관들이 실제로 합의하고 운영할 수 있는 **학습 인프라(구조)를 실증한 논문**이다.

## Background

### 왜 의료에서 협력 학습이 필요한가

의료 머신러닝은 환자 예후 예측, 질병 위험도 측정, 약물-결과 연관성 분석 등에 유용하지만, 단일 기관 데이터만으로는 표본 수가 부족하고 사망·합병증 같은 희귀 이벤트의 학습이 불안정하며 편향과 일반화 문제가 발생한다. 따라서 여러 의료기관의 데이터를 결합한 학습이 필요하다는 것이 대전제다.

데이터 통합에는 두 가지 현실적 시나리오가 있다.

- **Horizontal partition**: 여러 병원이 같은 변수 구조를 가지되 서로 다른 환자를 보유하는 경우. 기존 연합학습(FL)이 주로 다루던 상황이다.
- **Vertical partition**: 같은 환자에 대해 기관마다 서로 다른 정보(진료/검사, 처방, 장기 추적 기록 등)를 보유하는 경우. 의료 현실에서는 오히려 이쪽이 표준에 가깝다.

저자들은 기관을 hospital이 아니라 **site**라고 부르는데, site는 단순히 데이터가 있는 곳이 아니라 자기 데이터에 대한 소유권을 가지고 로컬 계산을 수행하며 전체 학습에 대등하게 참여하는 자율적 주체를 의미한다.

### 기존 접근법의 한계

- **Centralized learning**: 성능은 최고지만, 환자 데이터 이동으로 인한 재식별 위험 때문에 법·윤리·정책적으로 거의 불가능하다.
- **Federated learning**: 데이터를 로컬에 두어 프라이버시를 개선하지만, 중앙 서버가 필요하다. 이는 single point of control/failure가 되며, "누가 서버를 소유·운영하는가"라는 거버넌스 문제를 남긴다. 또한 대부분 horizontal만 지원하고 vertical 지원은 제한적이다.
- **기존 분산·암호화 기법**: horizontal 위주이며, 실전 의료 협력에 부적합하다.

저자들이 정의한 이상적 협력 학습 시스템의 조건은 네 가지 관점으로 정리된다. (1) **Data**: 수평·수직 통합을 모두 지원하되 환자 레벨 데이터는 절대 외부로 나가지 않을 것, (2) **Site**: 중앙 서버 없는 peer-to-peer 구조와 계산 부담의 형평성, (3) **Process**: 중앙집중 학습과 수학적으로 동등한 성능 및 학습 과정의 기록·감사 가능성, (4) **Model**: 모델 출처(provenance)의 검증 가능성과 대규모 모델 저장. 이 네 조건이 D-CLEF의 설계 철학이다.

## Method

### 시스템 아키텍처: 두 개의 네트워크

D-CLEF는 두 개의 permissioned 네트워크로 구성된다.

- **Modeling network (블록체인)**: 학습 순서, 파라미터 업데이트 이력, smart contract 실행을 기록한다. 블록체인에는 환자 수 $n$, covariate 수 $m$, site 정보, timestamp/iteration, 모델 타입·크기, 검증 결과, 실제 모델을 가리키는 content ID 같은 **메타데이터만** 저장된다. raw feature나 feature 이름·의미는 절대 올라가지 않는다.
- **Storage network (분산 파일 시스템)**: 압축된 모델 파라미터 자체를 저장한다. 블록체인에는 링크(hash)만 남기므로 감사 가능성과 확장성이 동시에 확보된다.

각 site는 로컬 데이터를 보관하는 computing node, 블록체인에 참여하는 modeling node, 모델을 공유하는 storage node라는 세 가지 논리적 역할을 동시에 수행한다. 요약하면 **데이터는 로컬, 과정은 블록체인, 결과물은 분산 스토리지**다.

### 학습 절차

D-CLEF에는 고정 서버가 없다. 대신 **round-robin** 방식으로 aggregator 역할이 site들 사이를 순환한다. 특정 기관이 항상 중앙이 되는 것을 구조적으로 막는 장치다.

로지스틱 회귀 기반의 vertical 학습 흐름을 보면 다음과 같다. 환자 $i$에 대해 각 site $k$는 자신의 feature block $x_i^{(k)}$만으로 partial logit을 계산한다.

$$z_i^{(k)} = x_i^{(k)} \cdot w^{(k)}$$

원본 feature는 전송하지 않고 이 partial logit만 이번 라운드의 aggregator site로 보낸다. Aggregator는 이를 단순 합산하여 예측 확률을 만든다.

$$z_i = \sum_{k} z_i^{(k)} + b, \qquad p_i = \sigma(z_i)$$

이후 공통 오차 신호 $e_i = p_i - y_i$를 계산해 모든 site에 브로드캐스트하고, 각 site는 자신의 파라미터만 로컬에서 업데이트한다.

$$\nabla w^{(k)} = \sum_i e_i \, x_i^{(k)}, \qquad w^{(k)} \leftarrow w^{(k)} - \eta \nabla w^{(k)}$$

이 업데이트는 중앙집중 로지스틱 회귀의 gradient descent와 **수학적으로 완전히 동일**하다. 어느 site도 다른 site의 feature를 끝까지 알 수 없으며, horizontal 시나리오에서는 partial logit 대신 로컬 gradient 요약을 집계하는 동일한 골격이 사용된다. 매 iteration이 끝나면 업데이트된 파라미터는 DFS에 저장되고 블록체인에 새 모델 hash가 기록되며 다음 aggregator가 지정된다.

## Experiments

세 가지 실제 의료 데이터셋으로 검증했다.

- **Dataset X — UC Health COVID-19 사망 예측**: 캘리포니아 대학(UC) 계열 5개 병원의 COVID-19 확진자 $n = 15{,}279$명, $m = 100$개 covariate(환자 정보 10, COVID 정보 6, 처방 약물 84). Vertical 시나리오를 위해 covariate를 (환자+COVID 정보 16개)와 (약물 84개) 두 블록으로 분할했다.
- **Dataset Y — 고관절 전치환술(THA) 후 장기 입원(3일 이상) 예측**: $n = 960$, $m = 34$. 환자를 층화 분할해 두 병원을 시뮬레이션하는 horizontal 실험과, covariate를 3개 블록으로 나눈 vertical 실험을 수행했다.
- **Dataset Z — 에든버러 심근경색 공개 데이터**: $n = 1{,}253$, $m = 9$ (증상 5개 + 심전도 4개). 증상 vs ECG로 나누는 vertical 구성이 자연스러운 데이터셋이다.

비교 대상은 (1) 각 병원이 자기 데이터로만 학습하는 **siloed learning**, (2) 모든 데이터를 모은 **centralized learning**(이론적 상한), (3) **D-CLEF**(horizontal/vertical)다.

핵심 결과는 일관적이다. 모든 병원에서 **siloed < D-CLEF ≈ centralized**가 관찰되었고, D-CLEF와 centralized 사이에는 통계적으로 유의미한 차이가 없었다. 즉 분산 구조라는 제약이 성능 저하로 이어지지 않으며, 성능 손실 없이 운영 구조를 바꿀 수 있음을 보여준다. 작은 초기 코호트에서 학습한 모델이 더 큰 후속 코호트에서도 안정적으로 유지되는지에 대한 시간적 검증에서도, D-CLEF는 중앙집중 학습을 잘 근사하면서 병원별 성능 편차도 작았다. 다만 기존 FL 대비 블록체인·DFS 동기화로 인한 시간 비용은 증가했다.

## Takeaways

이 논문이 증명한 것과 의도적으로 하지 않은 것을 구분하는 것이 중요하다.

**증명한 것**
- 중앙 서버 없이도 horizontal/vertical 모두에서 centralized와 동등한 성능이 가능하다.
- 기관 수 증가와 데이터 분포 변화에도 성능이 안정적이며, 특정 기관만 이득을 보는 구조가 아니다.
- 실제 다기관 의료 데이터에서 이를 입증했다.

**하지 않은 것**
- 새로운 VFL 알고리즘 제안, 암호학적 프라이버시 보장 증명, 딥러닝 모델 성능 비교, patient matching 문제 해결.

한계도 명확하다. 모델은 로지스틱 회귀 기반이고, feature engineering과 환자 매칭은 사전에 해결된 것으로 가정하며, 블록체인+DFS의 통신 비용은 실시간 의료 환경에 부담이 될 수 있다. 보안 위협 모델도 honest-but-curious — 프로토콜은 정직하게 따르지만 공유 정보로부터 다른 참여자의 데이터를 추론하려는 수동적 공격자 — 를 가정하며 악의적 공격은 다루지 않는다.

그럼에도 이 논문의 가치는 분명하다. 의료 협력의 실제 병목은 알고리즘이 아니라 **거버넌스와 신뢰**라는 점을 짚고, "누가 모델을 만들었고 언제 어떤 데이터 범위에서 학습되었는가"에 대한 시스템적 해답을 제시했다. 가장 빠른 FL이 아니라 **의료 기관들이 실제로 합의하고 운영할 수 있는 학습 인프라**를 만들었다는 것, 그것이 D-CLEF가 알고리즘 논문이 아니라 구조 논문인 이유다.
